### Systemic field gaps (review with editor)

These are missing for **every author in this batch**. Suggests an
ingest-pipeline gap, not 23 individual oversights:

- `scholar_id` (0 / 23)
- `orcid` (0 / 23)
- `email` (0 / 23)
- affiliation `role` and `start` (0 / 23)

Possible fixes: extend `ingest.py` to scrape Scholar profile + ORCID
from RePEc / personal pages; allow author-submission to populate
email; pull affiliation role/start from the institution faculty page
during photo hunt.

### `review_status` values not in SPEC

SPEC §2.2 only mentions `review_status` for Paper, where it's
`{ai-drafted, human-reviewed, author-approved}`. But the author JSONs
use `{confident, needs-verification, stub}`. That's a separate
vocabulary and should be either documented in SPEC §2.2 or aligned.

In this batch:
- 7 authors at `confident`
- 15 at `needs-verification`
- 0 at `stub` (per the handover, the 14 stub authors were rewritten
  in commit `69d497f` — confirmed: no stubs left in A–D).

### `qualifying_publication` is severely under-populated (3 / 23)

The 3 with it filled: berlinger-edina, danis-andras, domotor-barbara
(all `needs-verification` `review_status`, all under-13-fields). The
20 without are easy fills — the audit's proposed candidates use the
SPEC §1.2 tier ranking and the most-recent rule. Suggested workflow:
extend `bulk_audit.py` with a `--apply-qp` mode that writes the
proposal into each author JSON in one pass, then editor reviews via
`git diff`.

### Bisztray Márta is a clean baseline

## Open editorial questions surfaced by this audit

1. **SPEC §1.3 vs catalogue scope.** SPEC says "all peer-reviewed
   articles" for admitted authors, but several authors here have
   far fewer catalogue entries than RePEc would imply (Simonovits
   was the canonical example yesterday: 11 / 83). Either tighten
   the SPEC ("all peer-reviewed articles **in journals on
   `journals.json`**") or commit to a deeper backfill.

2. **Working papers in `data/papers/`.** Several papers in this
   batch have `publication_type: working_paper` in `data/papers/`
   (e.g. `simonovits-2018-wp`, `gabos-branyiczki-et-al-2018-wp`,
   `biro-branyiczki-2019-wp`, `adamecz-csillag-et-al-2022-wp`).
   These have no journal, no DOI in some cases. Per the
   2026-04-25 rule, only 2023+ NBER/CEPR/IZA/CESifo WPs are
   policy items. So WPs in `data/papers/` should be limited to
   "qualifying WP series, last 10 years". Cleanup pass might be
   worth running.

3. **Author `review_status` vocabulary.** Document in SPEC §2.2
   or rename to align with paper review_status (`stub` →
   `metadata-fetched`, `needs-verification` → ``ai-drafted`,
   `confident` → `human-reviewed`).

4. **`open_to_media_*` and `media_note`.** All 23 are at default
   (off). Worth a pass once the catalogue is more complete and the
   editor wants to start surfacing media-friendly authors.

## Skills validation notes

This batch used the `audit-author` skill and the new `bulk_audit.py`
helper script (file-level only, no web). Observations for refining
the skills:

1. **`audit-author/SKILL.md` should mention `bulk_audit.py`** — when
   the user asks for a batch ("audit X to Y"), use the script
   instead of calling the skill 23 times. Single-author invocations
   include the web-fetch external-coverage diff; bulk invocations
   skip it.

2. **Tier vocabulary** — `data/journals.json` uses tiers
   `{A, B, B-hu, C, ?}`, not just `{A, B}`. The skill mentions only
   A and B in §A. Add a note. The bulk script handles all five.

3. **Proposed `research-author --bulk`** — same pattern: a
   `bulk_research.py` helper that runs RePEc diff for all authors
   with `repec_id`, no web fetches beyond that, parallelisable.
   Useful as a standalone batch.

4. **`qualifying_publication` proposer was right almost every time.**
   Extending the helper to write directly to the author JSON (with
   a `--dry-run` default and a `--apply` flag) would close that gap
   in one pass.

5. **The audit caught one tier-mismatch:** `benk-szilard`'s
   highest catalogue tier paper is in *Economic Modelling*, which
   isn't on `journals.json`. So either he was admitted via the
   SPEC §1.2(ii) "≥3 published articles" path (which is fine), or
   his admission is borderline. Flag for editor.
